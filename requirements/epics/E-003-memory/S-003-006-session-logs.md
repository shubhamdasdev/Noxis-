## Story

As a **user**, I want **Noxis to keep a structured record of what we talked about each day**, so that **the daily brief can reference what's been happening in my life and the advice I receive builds on recent context without me having to recap it**.

## Goal

At the end of each day with active chat interactions, a session summary is generated and stored. The summary captures topics discussed, decisions made, actions committed, and new facts learned. This log feeds the daily brief generation engine (S-006-001) and long-term pattern detection. Logs are retained for 90 days then summarized to long-term memory.

## Why / Rationale

The orchestration pipeline uses the last 10 message pairs per conversation, which provides short-term continuity. But the daily brief and pattern detection need a higher-level view — what has this person been working on? What did they decide last Tuesday? Session logs bridge the gap between raw conversation history (too granular) and long-term memory (too sparse). They are the medium-term memory layer.

## Functional Requirements

- A session log is generated once per day per user if the user had at least one completed AI exchange that day
- Session log generation is triggered by a server-side scheduled job at 23:45 local time for each user (based on stored timezone); if the user has no messages that day, no log is created
- The session log is generated by an AI summarization call that receives all of the day's chat exchanges as input and produces a structured JSON summary:
  ```json
  {
    "date": "2026-03-18",
    "topics": ["outfit prep for a job interview", "protein intake for rest days"],
    "decisions": ["Going with the navy blazer", "Adding eggs back to breakfast"],
    "commitments": ["Will log every workout this week", "Will cook instead of ordering on weekdays"],
    "new_facts": ["User has a job interview on March 20", "User started intermittent fasting this week"],
    "module_activity": ["wardrobe", "food"],
    "message_count": 14
  }
  ```
- The structured JSON and a short prose summary (2-3 sentences in Noxis's voice) are both stored in the `session_logs` table
- Session logs are stored for 90 days; on day 91, a background archival job runs: the log's `new_facts` are passed through `MemoryService.addMemory` with `source: .session_log` and `confidence: 0.70`; the raw log record is then deleted
- If no messages were sent on a given day, no log entry is created — the date simply has no record
- The daily brief generation engine (S-006-001) reads session logs for the last 7 days as context input
- Session logs are never shown directly to the user

## Prerequisites

- S-003-001 (memory storage — archival job writes to Mem0)
- S-004-005 (orchestration pipeline — session logs require completed exchanges to summarize)

## Acceptance Criteria

- [ ] Given a user had 5 or more chat exchanges on a given day, when the scheduled job runs at 23:45 local time, then a session log entry is created for that date with all required JSON fields populated
- [ ] Given a user had zero chat exchanges on a given day, when the scheduled job runs, then no session log entry is created for that date
- [ ] Given a session log is created, when the JSON is inspected, then `topics`, `decisions`, `commitments`, `new_facts`, and `module_activity` are all present arrays (may be empty if the day had nothing extractable)
- [ ] Given a session log entry is 91 days old, when the archival job runs, then all `new_facts` from that log are written to Mem0 with `source: session_log` and the log record is deleted
- [ ] Given the daily brief generation engine requests context, when it queries session logs for the last 7 days, then all existing entries within that window are returned in date-descending order
- [ ] Given the summarization AI call fails for a day, when the job retries, then the job retries once after 5 minutes; if the second attempt fails, a `generation_failed` record is written for that date and no log entry is created

## Negative Scenarios

- User's timezone is not set (edge case during early onboarding) → the job falls back to UTC for scheduling; a warning is logged; timezone is inferred from the first message timestamp on the user's account
- Summarization AI call returns malformed JSON → the call is retried once; if the second attempt is also malformed, the prose summary is extracted as best-effort and the JSON fields are set to empty arrays; the partial log is still stored

## Edge Cases

- User sends messages across two timezones in one calendar day (e.g., flying) → the day boundary is determined by the user's stored timezone, not device timezone; all messages within the stored timezone's calendar day are included
- User has an extremely long conversation (200+ messages in one day) → the summarization call uses the last 50 message pairs as input to stay within context limits; earlier messages are excluded from that day's summary but their memory extractions (S-003-003) were already run in real time

## Dependencies

- S-003-001 (memory storage)
- S-004-005 (orchestration pipeline)

## Technical Requirements

- **`session_logs` table:** `(id UUID, user_id STRING, date DATE, summary_prose TEXT, summary_json JSONB, message_count INT, generated_at TIMESTAMP, status ENUM(generated, failed))`
- **Scheduled job:** `SessionLogGenerator` cron task — runs every 15 minutes; queries `users` for those whose local time is within the 23:45-23:59 window; processes each with a distributed lock to prevent double-run
- **Summarization prompt:** "Summarize this user's conversations from today. Return a JSON object with: topics (array of strings), decisions (array of strings), commitments (array of strings), new_facts (array of strings), module_activity (array of category names). Then write a 2-3 sentence prose summary in Noxis's voice of the day's key themes." Max tokens: 400.
- **Input to summarization:** all `ChatMessage` records for the user on the target date, sorted by `timestamp`; both user messages and Noxis responses included
- **90-day archival job:** `SessionLogArchiver` runs nightly (02:00 UTC); selects `session_logs WHERE date < NOW() - INTERVAL '90 days'`; for each, runs `MemoryService.addMemory` for each item in `summary_json.new_facts`; then deletes the record
- **Retry:** a `generation_failed` status triggers a single retry via the same cron task on its next pass; if still failing, status remains `failed`

## Future Scope

- User-accessible session log summaries (opt-in) — a "What did we talk about this week?" query in chat returns a natural language recap
- Pattern detection engine: reads 30 days of session logs to surface behavioral trends ("You've mentioned feeling tired 11 times this month — worth examining")
- Export: user can request a data export of all session logs as a PDF or text file

## Do Not Do

- In this story, do not display session logs to the user in any UI
- Do not use session logs as direct context in the orchestration pipeline — they feed into daily brief generation (S-006-001) and the archival memory path only
- Do not store raw message content in the session log — only the structured summary; raw messages live in the threads/messages tables
