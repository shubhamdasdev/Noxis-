## Story

As a **user**, I want **all the information I provided during onboarding to be immediately reflected in how Noxis talks to me**, so that **I never have to re-explain my basics — Noxis starts from where I left off**.

## Goal

When onboarding completes, every collected data point — tone preference, quick profile selections, conversational answers, and the first photo analysis — is structured into a USER.md schema and persisted as the user's core profile. This document becomes the persistent identity layer injected into every future conversation.

## Why / Rationale

Onboarding is the highest-intent moment in the user's relationship with Noxis. They just invested several minutes sharing real information. If that data evaporates and is not used to personalize the first real conversation, the product has failed at its most important first impression. USER.md is the bridge between onboarding and ongoing personalization.

## Functional Requirements

- When onboarding completes (after S-001-005 authentication), a `generateUserProfile` backend job is triggered
- The job reads all onboarding answers from the user's onboarding record in the database and assembles a structured USER.md document per the schema below
- **USER.md schema:**
  - `## Identity` — display name, age range (from quick profile), city/timezone (inferred or stated)
  - `## Tone Preference` — selected tone mode (brother / consultant / peer) with any qualifying notes from conversational onboarding
  - `## Physical` — height/weight range if stated, body goal (bulk/cut/maintain/general fitness), gym frequency (days/week)
  - `## Style` — fit preference (slim/regular/oversized), primary style descriptors (from quick profile selections), any noted brands or aesthetics from conversational responses
  - `## Food` — diet type (no restrictions/clean/intermittent fasting/other), delivery vs. cooking ratio if stated, any stated preferences or aversions
  - `## Spending` — income bracket (if stated), primary spend categories, any stated financial goals
  - `## Goals` — top 1-3 stated life areas the user wants to work on (from conversational onboarding)
  - `## First Impressions` — brief paragraph generated by the AI summarizing Noxis's read of this user based on all onboarding data combined; written in Noxis's voice
- Fields that were not answered during onboarding are marked `unset` rather than left blank or omitted
- USER.md is stored as a structured document in the backend database (`user_profile` table) alongside its raw text representation
- The USER.md raw text is injected into the orchestration pipeline as part of Step 4 (system prompt assembly) on every request
- When new memory facts are extracted (S-003-003), high-confidence core profile facts may trigger a USER.md field update (e.g., user mentions a new gym frequency → updates `## Physical` gym frequency field)
- USER.md is never shown directly to the user; it is a backend context document only

## Prerequisites

- S-001-003 (conversational onboarding — answers must exist before generation runs)
- S-001-005 (authentication — user account must exist to attach the profile to)
- S-003-001 (memory storage — initial profile facts are seeded to Mem0 after USER.md is generated)

## Acceptance Criteria

- [ ] Given onboarding completes and the user account is created, when `generateUserProfile` runs, then a USER.md document is created and stored within 5 seconds of account creation
- [ ] Given the user selected "Brother" tone during onboarding, when USER.md is generated, then the `## Tone Preference` section reflects "brother" as the selected mode
- [ ] Given the user answered conversational questions about their gym routine, when USER.md is generated, then the `## Physical` section contains the stated gym frequency and body goal
- [ ] Given onboarding did not collect an answer for a specific field (e.g., spending), when USER.md is generated, then that field is populated with `unset` — not omitted
- [ ] Given USER.md is generated, when the user sends their first real message after onboarding, then the orchestration pipeline includes the USER.md content in the system prompt and the response reflects onboarding context without the user having to restate it
- [ ] Given USER.md is generated, when the `## First Impressions` section is inspected, then it contains a non-empty paragraph written in Noxis's voice summarizing the user's overall profile
- [ ] Given a high-confidence memory is later extracted that contradicts a USER.md field, when the update logic runs, then the USER.md field is updated and the old value is stored as a previous value with a timestamp

## Negative Scenarios

- Onboarding data is partially complete (user skipped some screens) → `generateUserProfile` still runs; skipped fields are marked `unset`; the profile is usable with partial data
- AI call to generate the "First Impressions" paragraph fails → USER.md is stored without that section; the section is marked `pending_generation` and retried on the user's next app open

## Edge Cases

- User completes onboarding twice (e.g., by force-quitting during account creation and restarting) → second `generateUserProfile` run detects an existing profile; it merges new answers with the existing profile rather than overwriting — `unset` fields are filled in, confirmed fields are updated
- Onboarding conversational answers contain contradictions (e.g., user says they eat clean but then mentions they order food daily) → both facts are recorded in USER.md under their respective sections; the "First Impressions" paragraph acknowledges the apparent tension

## Dependencies

- S-001-003 (conversational onboarding)
- S-001-005 (authentication)
- S-003-001 (memory storage)

## Technical Requirements

- **`generateUserProfile(userId: String)` (backend)** — reads `onboarding_responses` table for the user; maps answers to USER.md schema fields; calls AI provider to generate `## First Impressions` paragraph; writes result to `user_profiles` table as `{ user_id, content: String, version: Int, created_at, updated_at }`
- **USER.md as text:** stored in `user_profiles.content` as a plain Markdown string; also stored as structured JSON in `user_profiles.structured_data` for field-level updates
- **First Impressions generation:** AI call with prompt: "Based on this onboarding data, write a brief, direct, first-person-perspective summary of this user in Noxis's voice (brother/consultant/peer tone depending on their selection). 2-3 sentences. Be specific to their actual answers." Max tokens: 200.
- **Mem0 seed:** after USER.md is written, `MemoryService.addMemory` is called for each non-`unset` field with confidence 1.0 and `source: .onboarding`
- **USER.md update trigger:** `MemoryService.addMemory` emits a `MemoryAdded` event; a `UserProfileUpdater` listener checks if the new memory matches a USER.md field pattern; if confidence > 0.85 and the field currently differs, it applies the update
- **Pipeline injection:** `OrchestrationPipeline` fetches `user_profiles.content` on each run; injects between tone instruction (layer 2) and module context (layer 3)

## Future Scope

- User-visible profile summary screen (a curated, simplified view of USER.md without exposing the raw document)
- USER.md diff view in admin/debug panel showing changes over time
- Manual profile correction flow where user can adjust specific fields if Noxis has learned something incorrectly

## Do Not Do

- In this story, do not show USER.md to the user in the UI
- Do not implement the memory review screen — that is S-003-007
- Do not implement automatic memory extraction from ongoing chat — that is S-003-003; this story only handles the initial generation
